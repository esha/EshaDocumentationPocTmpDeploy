Remarks
 

A "root" or "aggregate root" entity is an entity which can be queried on via a repository and manages its own lifespan and optionally the lifespan of associated entities which are non-root entities. An example in the ESHA domain is a Food entity and a ConversionMap. A ConversionMap doesn't have a meaning outside of the food it is associated with, and therefore doesn't have a repository which maintains an independent collection of them. Each ConversionMap is bound to the Food or NutrientProfile which contains it. Therefore, entities like Food are a RootEntity and ConversionMap is not.

From "Domain Driven Design" by Eric Evans: "For each type of object that needs global access, create an object that can provide the illusion of an in-memory collection of all objects of that type. Set up access through a well-known global interface. Provide methods to add and remove objects, which will encapsulate the actual insertion and removal of data in the data store. Provide methods that select objects based on some criteria and return fully instantiated objects or collections of objects whose attribute values meet the criteria, thereby encapsulating the actual storage and query technology. Provide repositories only for aggregate roots that actually need direct access. Keep the client focused on the model, delegating all object storage and access to the Repositories.
